# ABYSSAC — 记忆系统

## 核心原则

**静默执行**：检索、存储全部后台完成，不向用户展示操作过程。
**直接回答**：只输出对用户问题的回答，不解释操作过程。
**强制完整**：记忆文件必须存完整内容，禁止只存摘要。
**Pipeline不可跳步**：每步有明确门控条件，未通过不得进入下一步。

---

## 存储根路径

```
{ABYSSAC根路径}/
├── nng/
│   ├── root.json
│   ├── {摘要}.json
│   └── {摘要}/{摘要}.json
├── Y层记忆库/
│   ├── 元认知记忆/{year}/{month}/{day}/{id}.txt
│   ├── 高阶整合记忆/{year}/{month}/{day}/{id}.txt
│   └── 分类记忆/{高|中|低}/{year}/{month}/{day}/{id}.txt
└── ProjectCache/
    └── {项目名}.cache.md
```

---

## 记忆类型

| 类型 | 存放内容 |
|------|----------|
| 元认知记忆 | 推理策略、系统性错误、能力边界、有效解题模式 |
| 高阶整合记忆 | 跨领域知识整合、综合性结论 |
| 分类记忆 | 具体领域知识、用户偏好、项目决策、技术细节 |

**价值层级：** 高（置信度≥90）/ 中（70-89）/ 低（<70）

---

## 项目缓存（ProjectCache）

**设计原则：**
- 只读快照，由检索Pipeline生成，后续对话直接注入
- 项目隔离，一项目一文件，互不干扰
- 严禁直接写入，新知识只走存储Pipeline写NNG体系
- 按需重新生成，缓存标记待更新后，语义判断是否涉及新知识才重新生成

**缓存文件格式：**
```
# ProjectCache: {项目名}
生成时间: {ISO8601}
ABYSSAC版本: 4.1
状态: 正常 | 待更新

## 记忆内容

### [{NNG节点摘要}]
（置信度: {0-100}）
{关联记忆内容，按置信度降序}
{instruction规则内嵌在记忆内容后面，每条单独一行}
- IF {条件} THEN {行动}
...

## NNG路径索引

nng/
├── {一级节点}.json
│   └── {一级节点}/
│       ├── {二级节点}.json
│       └── ...
├── {一级节点}.json
│   └── ...
└── ...
```

**生命周期：**
```
项目启动
├─ 缓存存在且状态正常 → 直接加载，跳过检索
├─ 缓存存在且状态待更新 → 语义判断是否涉及新知识
│   ├─ 是 → 重新运行检索Pipeline，生成新缓存
│   └─ 否 → 直接加载旧缓存
└─ 缓存不存在 → 运行检索Pipeline，生成缓存

对话中产生新知识 → 存储Pipeline写NNG体系，缓存标记 状态: 待更新
```

**失效条件：**

| 条件 | 处理 |
|------|------|
| 用户主动要求刷新 | 立即删除缓存，重新检索 |
| 存储了新知识 | 标记待更新，下次按需重新生成 |
| 切换项目 | 加载对应项目缓存，不触发重新生成 |

**NNG路径索引用途：**
重新生成缓存时优先读取索引中的节点，跳过无关遍历；同时可辅助调试定位知识来源。

**项目名约定：**
由用户声明或从首条消息提取，格式 `{项目名}.cache.md`，同名复用。
# ABYSSAC — 执行Pipeline

## 执行入口

```
├─ 缓存存在 → 见ProjectCache生命周期
├─ 对话产生有价值的新知识 → 执行存储Pipeline
└─ 都不涉及 → 直接回答
```
不确定是否需要检索时，默认执行检索。

---

## 检索Pipeline

> 每步门控：无结果则记录并跳出，不得伪造匹配。检索完成后必须生成ProjectCache。

### R1 — 遍历root.json
```
1. 读取 {ABYSSAC根路径}/nng/root.json
2. 提取问题关键词 + 理解语义意图
3. 对每个一级节点逐一语义判断：与当前问题有关联吗？→ 是/否，不确定默认"是"
4. 记录所有"是"的节点
【门控】读取成功 → R2 / 不存在 → 跳出直接回答
```

### R2 — NNG深入遍历
```
对R1判断为"是"的节点：
1. 读取节点，检查"已更新"字段，存在则跳转新节点
2. 对每个下级NNG语义判断：与当前问题有关联吗？→ 是则递归，否则跳过
3. 优先扫描元认知记忆对应节点
【门控】至少一个节点成功 → R3 / 全部失败 → 跳出直接回答
```

### R3 — 记忆筛选加载
```
对每个NNG节点的关联记忆：
1. 检查load_rule：匹配when → 读取 / 匹配not_when → 跳过 / 都不匹配 → 默认读取
2. 读取记忆文件，执行instruction中匹配的IF条件
3. 按置信度排序：高≥90 > 中70-89 > 低<70
【门控】至少一条成功 → R4 / 无记忆 → 跳出直接回答
```

### R4 — 组装缓存
```
1. 元认知节点优先，其余按命中顺序排列，每节点下记忆按置信度降序
2. 从每个命中的NNG节点中读取instruction规则，写入缓存对应节点下
3. 写入 {ABYSSAC根路径}/ProjectCache/{项目名}.cache.md
4. 以缓存为上下文生成回答
```

## 存储Pipeline

> 只写NNG体系和Y层记忆库，严禁修改缓存文件。

### S1 — 价值判断
```
存储（满足任一）：✅用户要求记住 ✅新知识/规则 ✅用户偏好 ✅项目决策 ✅技术细节
不存储：❌寒暄 ❌完全重复 ❌无实质信息过渡语句
【门控】满足 → S2 / 不满足 → 跳出
```

### S2 — 重复检查
```
1. 检查root.json是否有相同摘要节点
2. 若有，检查内容字段是否高度相似
3. 检查记忆文件是否有相同核心内容
完全重复 → 跳出 / 部分重复 → S3新增关联 / 矛盾过时 → S3更新 / 全新 → S3创建
【门控】无完全重复 → S3
```

### S3 — 分类路径
```
类型：推理策略/错误/边界 → 元认知 / 跨领域 → 高阶整合 / 其他 → 分类
层级：核心/强调 → 高(≥90) / 一般 → 中(70-89) / 辅助 → 低(<70)
ID：YYYYMMDDHHmmss + 3位序号
路径：Y层记忆库/{类型}/{层级}/{year}/{month}/{day}/{id}.txt
```

### S4 — 记忆文件
```
记忆层级: {元认知/高阶整合/分类}
记忆ID: {id}
记忆时间: {ISO8601}
置信度: {0-100}
核心内容: |
  {完整内容，多行缩进}
instruction:
- IF {具体条件} THEN {明确行动}
关联NNG: {摘要}:{路径}
标签: {标签1}, {标签2}
价值评估: {高/中/低}
```

### S5 — NNG节点
**新建：**
```json
{
  "定位": "{摘要5-20字}", "置信度": {0-100}, "时间": "{ISO8601}",
  "内容": "{核心内容≤100字}",
  "instruction": ["IF {条件} THEN {行动}"],
  "load_rule": {"when": ["{场景}"], "not_when": ["{场景}"]},
  "上级关联NNG": [{"节点ID": "{摘要}", "路径": "nng/{摘要}.json", "关联程度": {0-100}}],
  "下级关联NNG": [{"节点ID": "{摘要}", "路径": "nng/{摘要}/{摘要}.json", "关联程度": {0-100}}],
  "关联的记忆文件摘要": [{"记忆ID": "{id}", "路径": "Y层记忆库/...", "摘要": "{20-50字}", "记忆类型": "{类型}", "价值层级": "{高/中/低}", "置信度": {0-100}}]
}
```
**更新旧节点：**
```json
{
  "定位": "{原摘要}", "置信度": "{原}", "时间": "{原}", "内容": "{原}",
  "已更新": {"状态": true, "更新时间": "{ISO8601}", "更新原因": "{说明}", "新节点路径": "nng/{新摘要}.json"},
  "关联的记忆文件摘要": []
}
```

### S6 — 更新索引
```
一级节点 → 更新root.json，格式："{摘要}:{路径}:{内容简述}"
下级节点 → 在父节点"下级关联NNG"中添加引用
完成后 → 缓存头部标记 状态: 待更新
```
root.json: `{"一级节点": ["{摘要}:{路径}:{内容简述}"], "更新时间": "{ISO8601}"}`

## Reviewer（每次操作后必须运行）

### 检索验证
```
🔴 root.json是否已读取？
🔴 一级节点是否逐一语义判断？（不得批量跳过）
🔴 "已更新"节点是否已跳转？
🔴 是否已生成ProjectCache？
🔴 存储了新知识则标记缓存待更新？
🟡 下级节点是否逐一语义判断？
🟡 load_rule都不匹配时是否默认读取？
🟡 元认知节点是否优先扫描？
🟢 记忆是否按置信度排序？
🟢 回答中是否未包含操作过程？
```

### 存储验证
```
🔴 是否执行重复检查？
🔴 记忆文件是否包含完整内容？
🔴 NNG与记忆是否双向关联？
🔴 root.json是否已更新？
🔴 记忆ID是否唯一？
🔴 是否未修改缓存文件？
🟡 instruction是否具体可执行？
🟡 价值层级与置信度是否一致？
🟡 父节点下级关联是否已更新？
🟢 标签是否准确？
🟢 load_rule是否覆盖主要场景？
```

---

## instruction规范

格式：`IF {具体条件} THEN {明确行动}`
✅ `IF 用户询问API参数 THEN 读取记忆文件提供完整参数列表`
❌ `IF 需要时 THEN 读取`（条件模糊）

---

## NNG层级

| 级别 | 路径 |
|------|------|
| 一级 | `nng/{摘要}.json` |
| 二级 | `nng/{一级}/{二级}.json` |
| 三级+ | `nng/{一级}/{二级}/{三级}.json` |

导航链超3层时，ProjectCache的NNG路径索引已记录命中节点，下次优先读取跳过中间层级。

---

## 原则汇总

1. 不确定是否检索时，默认检索
2. load_rule都不匹配时，默认读取（激进策略）
3. Pipeline不可跳步，门控不满足则跳出
4. 存储前必须执行重复检查
5. 记忆存完整内容，NNG只存摘要索引
6. NNG与记忆必须双向关联
7. 不删除记忆，只降低置信度
8. 对话结束后立即存储
9. 每次操作后必须运行Reviewer，发现问题立即修正
10. ProjectCache只由检索Pipeline生成，存储Pipeline严禁写入
11. 项目启动优先检查缓存，有缓存则直接加载
13. 缓存元认知优先，其余按命中顺序，同节点下按置信度降序
