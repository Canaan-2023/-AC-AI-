# ABYSSAC — 记忆系统

## 核心原则

**静默执行**：检索、存储全部后台完成，不向用户展示任何操作过程。  
**直接回答**：最终只输出对用户问题的回答，不解释"我在检索"或"我已存储"。  
**强制完整**：记忆文件必须存完整内容，禁止只存摘要。  
**Pipeline不可跳步**：检索流程和存储流程均为严格顺序，每步有明确门控条件，未通过门控不得进入下一步。

## 存储根路径

```
d:\ABYSSAC/
├── nng/
│   ├── root.json                         # 一级节点索引
│   ├── {摘要}.json                        # 一级节点
│   └── {摘要}/
│       └── {摘要}.json                    # 二至N级节点
├── Y层记忆库/
│   ├── 元认知记忆/{year}/{month}/{day}/{id}.txt
│   ├── 高阶整合记忆/{year}/{month}/{day}/{id}.txt
│   └── 分类记忆/{高|中|低}/{year}/{month}/{day}/{id}.txt
└── ProjectCache/
    └── {项目名}.cache.md                  # 项目记忆快照（ABYSSAC运行后生成）
```

## 记忆类型

| 类型 | 存放内容 | 优先级 |
|------|----------|--------|
| 元认知记忆 | 关于自身认知过程的知识：哪类问题适合什么推理策略、过去犯过的系统性错误、对自身能力边界的认知、某类任务的有效解题模式 | 最高，优先扫描 |
| 高阶整合记忆 | 跨领域知识整合、综合性结论 | 次高 |
| 分类记忆 | 具体领域知识、用户偏好、项目决策、技术细节 | 按价值层级排序 |

**价值层级与置信度：**

| 层级 | 置信度 | 检索优先级 |
|------|--------|------------|
| 高 | ≥ 90 | 1（优先） |
| 中 | 70–89 | 2 |
| 低 | < 70 | 3（最后） |

## 项目缓存文件（ProjectCache）

### 设计原则

ProjectCache是ABYSSAC检索Pipeline的**输出产物**，不是独立维护的数据库。

- **只读快照**：由ABYSSAC运行后生成，后续对话直接注入，不需要每次重跑检索Pipeline
- **项目隔离**：一个项目对应一个缓存文件，不同项目互不干扰
- **禁止直接写入**：新知识产生时，只走存储Pipeline写入NNG体系，**严禁直接追加到缓存文件**
- **重新生成**：下次启动项目时重新运行ABYSSAC检索，生成新缓存（此时新知识自然被包含）

### 缓存文件格式

```markdown
# ProjectCache: {项目名}
生成时间: {ISO8601时间}
ABYSSAC版本: 4.2
状态: 正常 | 待更新

---

## 记忆内容
（按NNG检索顺序排列：匹配程度高的节点及其关联记忆在前）

### [{NNG节点摘要}] 匹配程度: {分数}
{该节点关联的记忆内容，按置信度降序}

### [{NNG节点摘要}] 匹配程度: {分数}
{该节点关联的记忆内容，按置信度降序}

...

---

## NNG路径索引
{本次检索命中的NNG节点路径列表，按匹配程度降序，供下次生成参考}
```

### 缓存的生命周期

```
项目启动
    │
    ├─ 存在 {项目名}.cache.md？
    │   ├─ 是 → 直接加载缓存，跳过检索Pipeline，进入对话
    │   └─ 否 → 执行检索Pipeline → 生成缓存文件 → 进入对话
    │
对话中产生新知识
    │
    └─ 执行存储Pipeline → 只写入NNG体系和Y层记忆库
                          禁止写入缓存文件
    │
项目下次启动
    │
    └─ 重新运行检索Pipeline → 生成新缓存（包含上次存入的新知识）
```

### 缓存失效机制

缓存失效后必须删除旧缓存文件，下次启动时重新运行检索Pipeline生成新缓存。

| 失效条件 | 触发方式 | 处理 |
|----------|----------|------|
| 用户明确要求刷新记忆 | 主动声明 | 立即删除缓存，重新检索生成新缓存 |
| 本次对话存储了新知识 | 存储Pipeline执行完成后在缓存头部写入 `状态: 待更新` | 当次继续用旧缓存；下次加载时判断是否需要更新 |

**待更新缓存的判断逻辑**：

```
加载缓存，发现 状态: 待更新
    │
    ├─ 用户当前问题涉及新存入的知识？
    │   ├─ 是 → 重新运行检索Pipeline，生成新缓存，再回答
    │   └─ 否 → 继续使用旧缓存，不重新生成
```

也就是说，缓存是否重新生成取决于"当前是否真的需要更新的记忆"，而不是一律在下次启动时重新生成。

**项目隔离**：

每个项目维护独立的缓存文件，项目之间完全隔离。切换项目只是加载对应的缓存文件，不触发任何重新生成。各项目缓存只受自身项目内的知识变动影响，互不干扰。

### NNG路径索引的用途

缓存文件末尾的"NNG路径索引"记录本次检索命中的所有NNG节点路径，用途有二：

**加速下次缓存生成**：重新生成缓存时，优先直接读取索引中的节点，跳过root.json中无关节点的遍历，减少检索开销。但仍须验证这些节点未被标记"已更新"，若有则跳转新节点。

**辅助调试**：可通过路径索引快速定位本次缓存的知识来源，判断检索是否遗漏了相关节点。

格式示例：
```
## NNG路径索引
nng/ABYSSAC开发.json
nng/ABYSSAC开发/NNG设计.json
nng/用户偏好.json
```

### 项目名约定

- 项目名由用户在对话开始时声明，或由第一条消息的主题自动提取
- 文件名格式：`{项目名}.cache.md`，例如 `ABYSSAC开发.cache.md`
- 项目名相同视为同一项目，复用同一缓存文件

## 执行入口判断

```
每次对话开始时：

├─ 是否存在当前项目的缓存文件？
│   ├─ 是，且状态正常 → 直接加载，跳过检索Pipeline
│   ├─ 是，且状态为待更新 → 判断当前问题是否涉及新知识
│   │   ├─ 是 → 重新运行检索Pipeline，生成新缓存
│   │   └─ 否 → 直接加载旧缓存
│   └─ 否 → 运行检索Pipeline，生成缓存
│
├─ 对话产生了有价值的新知识？
│   └─ 是 → 执行存储Pipeline（只写NNG体系，缓存头部标记待更新）
│
└─ 都不涉及 → 直接回答
```

**项目隔离**：不同项目各自维护独立缓存，切换项目只是加载对应文件，不触发重新生成。

## 检索Pipeline（Tool Wrapper + Pipeline模式）

> **门控规则：每步必须有结果才能进入下一步，无结果时记录并跳出，不得伪造匹配。**
> **检索Pipeline完成后必须生成ProjectCache文件。**

### Step R1 — 强制遍历root.json【门控：必须读取文件后才能继续】

```
1. 读取 d:\ABYSSAC\nng\root.json
2. 提取用户问题的关键词 + 理解语义意图
3. 对root.json中每个一级节点，逐一评估：
   - 关键词匹配 或
   - 语义相关（概念层面有关联）
4. 记录所有匹配节点（匹配程度用0-100评分）

【门控】：root.json读取成功 → 进入R2
        root.json不存在 → 跳出检索流程，直接回答，不生成缓存
```

### Step R2 — NNG节点深入遍历【门控：有匹配节点才继续】

```
对Step R1中匹配程度 ≥ 60 的节点：
1. 读取该节点文件
2. 检查"已更新"字段：
   - 存在 → 自动跳转新节点路径，读取新节点
   - 不存在 → 继续读取当前节点
3. 查看下级关联NNG：
   - 关联程度 ≥ 70 → 递归读取下级节点
   - 关联程度 < 70 → 不深入
4. 优先扫描元认知记忆对应的NNG节点

【门控】：至少一个节点读取成功 → 进入R3
        所有节点读取失败 → 跳出，直接回答，不生成缓存
```

### Step R3 — 记忆筛选与加载【门控：有关联记忆路径才继续】

```
对每个NNG节点中的关联记忆：
1. 检查load_rule：
   - 匹配 when 条件 → 读取完整记忆文件
   - 匹配 not_when 条件 → 跳过
   - 两者都不匹配 → 默认读取（激进策略，宁多勿漏）
2. 读取记忆文件
3. 执行记忆中的 instruction（IF条件匹配则执行对应行动）
4. 按置信度排序：高≥90 > 中70-89 > 低<70

【门控】：至少一条记忆读取成功 → 进入R4
        无记忆可读 → 跳出，直接回答，不生成缓存
```

### Step R4 — 上下文组装 + 生成缓存

```
1. 按NNG匹配程度降序排列所有命中节点
2. 每个节点下的记忆按置信度降序排列
3. 按缓存文件格式写入 d:\ABYSSAC\ProjectCache\{项目名}.cache.md
4. 以缓存内容为上下文，生成对用户问题的回答（不包含任何检索过程描述）
```

## 存储Pipeline（Pipeline + Reviewer模式）

> **门控规则：存储前必须通过重复检查，重复内容禁止创建新节点。**
> **存储Pipeline只写NNG体系和Y层记忆库，严禁修改缓存文件。**

### Step S1 — 知识价值判断【门控：有价值才继续】

```
存储触发条件（满足任一即存储）：
✅ 用户明确要求记住
✅ 新的知识/规则/方法论
✅ 用户偏好信息
✅ 项目相关决策
✅ 技术细节（API、配置、参数等）

不存储：
❌ 寒暄问候
❌ 已存在的完全相同内容
❌ 无实质信息的过渡语句

【门控】：满足存储条件 → 进入S2
        不满足 → 跳出存储流程
```

### Step S2 — 重复检查【门控：通过去重才继续】

```
1. 检查root.json，是否已有相同摘要的一级节点？
2. 若有，读取该节点，检查"内容"字段是否与新知识相同或高度相似？
3. 检查相关记忆文件，是否已有相同核心内容？

结果处理：
- 完全重复 → 跳出，不创建
- 部分重复（新信息补充旧信息）→ 进入S3，新增节点并建立关联
- 矛盾/过时 → 进入S3，更新旧节点
- 全新内容 → 进入S3，创建新节点

【门控】：确认无完全重复 → 进入S3
```

### Step S3 — 确定分类与路径

```
1. 判断记忆类型：
   - 关于自身推理策略、系统性错误、能力边界的认知 → 元认知记忆
   - 跨领域整合 → 高阶整合记忆
   - 其他 → 分类记忆

2. 判断价值层级：
   - 核心知识、用户强调的内容 → 高（置信度≥90）
   - 一般知识 → 中（置信度70-89）
   - 辅助信息 → 低（置信度<70）

3. 生成记忆ID（格式：YYYYMMDDHHmmss + 3位序号，如202503261030001）

4. 确定文件路径：
   Y层记忆库/{类型}/{价值层级}/{year}/{month}/{day}/{id}.txt
```

### Step S4 — 创建记忆文件

```
按以下模板创建完整记忆文件（禁止只存摘要）：

记忆层级: {元认知/高阶整合/分类}
记忆ID: {id}
记忆时间: {ISO8601时间}
置信度: {0-100}
核心内容: |
  {完整内容，包含所有细节，多行使用缩进}
instruction:
- IF {具体条件} THEN {明确行动}
- IF {具体条件} THEN {明确行动}
关联NNG: {摘要}:{路径}
标签: {标签1}, {标签2}, {标签3}
价值评估: {高/中/低}
```

### Step S5 — 创建或更新NNG节点

**新建NNG节点时：**

```json
{
  "定位": "{简要摘要，5-20字}",
  "置信度": {0-100},
  "时间": "{ISO8601时间}",
  "内容": "{节点核心内容，简短，不超过100字}",
  "instruction": [
    "IF {具体条件} THEN {明确行动}"
  ],
  "load_rule": {
    "when": ["{需要读取详情的场景}"],
    "not_when": ["{明确不需要读取的场景}"]
  },
  "上级关联NNG": [
    {
      "节点ID": "{摘要}",
      "路径": "nng/{摘要}.json",
      "关联程度": {0-100}
    }
  ],
  "下级关联NNG": [
    {
      "节点ID": "{摘要}",
      "路径": "nng/{摘要}/{摘要}.json",
      "关联程度": {0-100}
    }
  ],
  "关联的记忆文件摘要": [
    {
      "记忆ID": "{id}",
      "路径": "Y层记忆库/{类型}/{价值}/{year}/{month}/{day}/{id}.txt",
      "摘要": "{记忆内容摘要，20-50字}",
      "记忆类型": "{类型}",
      "价值层级": "{高/中/低}",
      "置信度": {0-100}
    }
  ]
}
```

**更新旧NNG节点时（内容矛盾或过时）：**

```json
{
  "定位": "{原摘要}",
  "置信度": "{原置信度}",
  "时间": "{原创建时间}",
  "内容": "{原内容}",
  "已更新": {
    "状态": true,
    "更新时间": "{ISO8601时间}",
    "更新原因": "{简要说明}",
    "新节点路径": "nng/{新摘要}.json"
  },
  "关联的记忆文件摘要": []
}
```

### Step S6 — 更新索引

```
1. 更新 d:\ABYSSAC\nng\root.json：
   - 新增一级节点条目（如为一级节点）
   - 格式："{摘要}:{路径}:{内容简述}"

2. 更新父节点（如为二级及以下节点）：
   - 在父节点的"下级关联NNG"中添加新节点引用

3. root.json格式：
{
  "一级节点": [
    "{摘要}:{路径}:{内容简述}",
    "{摘要}:{路径}:{内容简述}"
  ],
  "更新时间": "{ISO8601时间}"
}
```

## 执行验证Reviewer（必须在每次执行后运行）

> **这是强制步骤。每次完成检索或存储操作后，必须逐项检查以下清单，发现问题立即修正，修正后再生成回答。**

### 存储验证清单

```
存储完成后逐项确认：

🔴 严重（任一不通过，必须修正后才能完成）
□ 是否执行了重复检查？
□ 记忆文件是否包含完整内容（非摘要）？
□ NNG节点与记忆文件是否双向关联？
□ root.json是否已更新？
□ 记忆ID是否唯一？
□ 是否未修改缓存文件？（存储Pipeline严禁写缓存）

🟡 中等（不通过，记录并尝试修复）
□ instruction是否具体可执行（非模糊描述）？
□ 价值层级与置信度是否一致（高≥90/中70-89/低<70）？
□ 父节点的下级关联是否已更新？

🟢 低（不通过，下次改进）
□ 标签是否准确描述内容？
□ load_rule的when/not_when是否覆盖主要使用场景？
```

## instruction编写规范

**格式**：`IF {具体条件} THEN {明确行动}`

**好的示例：**
```
IF 用户询问该API的具体参数 THEN 读取关联记忆文件并提供完整参数列表
IF 用户要修改该配置 THEN 读取记忆文件并提示当前配置值
IF 用户确认该信息存在即可 THEN 不读取详细记忆文件，直接确认
```

**禁止的写法：**
```
❌ IF 需要时 THEN 读取（条件模糊）
❌ IF 用户问 THEN 回答（行动不明确）
```

**原则**：一条instruction对应一个明确的条件-行动对，条件可直接判断，行动可直接执行。

## NNG节点层级说明

| 级别 | 路径 | 作用 |
|------|------|------|
| 一级 | `nng/{摘要}.json` | 大的知识领域或概念分类 |
| 二级 | `nng/{一级摘要}/{二级摘要}.json` | 一级节点下的具体概念 |
| 三级+ | `nng/{一级}/{二级}/{三级摘要}.json` | 更细分的概念 |

**元认知NNG**：当导航链超过3层时，ProjectCache的NNG路径索引已记录命中节点，下次重新生成缓存时直接优先读取这些节点，跳过无关遍历。

## 原则汇总

1. 静默执行，不输出思维过程
2. 不确定是否需要检索时，默认检索
3. load_rule两者都不匹配时，默认读取（激进策略）
4. Pipeline各步骤不可跳过，门控条件不满足则记录并跳出
5. 存储前必须执行重复检查
6. 记忆必须存完整内容，NNG只存摘要索引
7. NNG与记忆必须双向关联
8. 不删除记忆，只降低置信度（降权）
9. 对话结束后立即存储，无需后台任务
10. 每次操作后必须运行Reviewer验证清单，发现问题立即修正
11. ProjectCache只由检索Pipeline生成，存储Pipeline严禁写入缓存
12. 项目启动时优先检查缓存，有缓存则直接加载，跳过检索Pipeline
13. 缓存文件按NNG匹配程度降序排列，同一节点下的记忆按置信度降序排列
